# Requirements Document

## Introduction

The Recruiter RAG Platform is a web application that enables recruiters to search for candidates using natural language or job descriptions. It is powered by a Retrieval-Augmented Generation (RAG) pipeline over candidate profiles collected from the AI Job Assistant Chrome extension. Candidate profiles are only searchable when the candidate has explicitly consented (`shareAnonymized=true`). PII is masked by default and revealed only to authenticated users who spend an event credit. The first search is free and requires no login; subsequent searches and PII unmasking require Supabase Google OAuth sign-in.

## Glossary

- **Recruiter**: A user of the web application who searches for candidates.
- **Candidate**: A job seeker whose profile was collected by the AI Job Assistant Chrome extension.
- **Profile**: A structured document containing a candidate's skills, experience, location, role/title, and PII (name, phone, email).
- **PII**: Personally Identifiable Information — name, phone number, and email address.
- **RAG_Pipeline**: The Retrieval-Augmented Generation system that embeds a job description, performs vector search, and uses an LLM to generate ranking explanations.
- **Embedding_Service**: The Ollama `nomic-embed-text` model used to generate vector embeddings.
- **LLM**: The Ollama `llama3.2:1b` model used to generate AI ranking explanations.
- **pgvector**: The PostgreSQL extension used to store and query vector embeddings.
- **Event**: A single billable action — either unmasking PII for one candidate or downloading a resume.
- **Event_Counter**: The per-user count of events consumed, tracked in Supabase.
- **Free_Tier**: The allowance of events available to a logged-in user at no cost, configurable via `MAX_FREE_EVENTS` environment variable (default: 10).
- **Match_Score**: A numeric relevance score (0–100) produced by the vector similarity search.
- **AI_Reasoning**: A short natural-language explanation generated by the LLM describing why a candidate matches the job description.
- **Supabase_Auth**: The authentication service using Google OAuth, provided by Supabase.
- **Extension**: The AI Job Assistant Chrome extension that collects candidate profiles.
- **Backend**: The FastAPI service deployed on Railway.
- **Frontend**: The React web application deployed on Railway or Vercel.

## Requirements

### Requirement 1: Candidate Profile Sync from Extension

**User Story:** As a candidate using the AI Job Assistant extension, I want my profile to be saved to the recruiter platform when I consent, so that recruiters can discover me.

#### Acceptance Criteria

1. WHEN the Extension POSTs a profile to `POST /api/v1/profile/sync` with `shareAnonymized=true`, THE Backend SHALL embed the profile text using the Embedding_Service and store the embedding and profile data in pgvector.
2. IF a profile is received with `shareAnonymized=false` or the field is absent, THEN THE Backend SHALL reject the profile with HTTP 403 and not store any data.
3. WHEN a profile is synced for a candidate whose profile already exists (matched by a stable candidate hash), THE Backend SHALL update the existing record rather than create a duplicate.
4. THE Backend SHALL store the following fields per profile: candidate hash, role/title, skills list, location, and masked PII (name, phone, email stored encrypted or separately).
5. IF the Embedding_Service is unavailable during a sync request, THEN THE Backend SHALL return HTTP 503 with a descriptive error message.

---

### Requirement 2: Job Description Search

**User Story:** As a recruiter, I want to paste a job description and get a ranked list of matching candidates, so that I can quickly identify the best fits.

#### Acceptance Criteria

1. WHEN a recruiter submits a job description to `POST /api/v1/recruiter/search`, THE RAG_Pipeline SHALL embed the job description using the Embedding_Service and perform a vector similarity search against stored candidate profiles in pgvector.
2. THE RAG_Pipeline SHALL return the top 20 candidates ranked by Match_Score in descending order.
3. WHEN search results are returned, THE Backend SHALL include for each candidate: rank, role/title, skills, location, Match_Score, and AI_Reasoning.
4. THE Backend SHALL generate AI_Reasoning for each returned candidate using the LLM.
5. WHEN search results are returned, THE Backend SHALL mask all PII fields (name, phone, email) in the response, replacing each with a placeholder string (e.g., `"****"`).
6. WHEN a recruiter has not yet performed any search in the current session (first search), THE Backend SHALL process the search request without requiring authentication.
7. WHEN a recruiter attempts a second or subsequent search without being authenticated, THE Backend SHALL return HTTP 401 with a message indicating sign-in is required.
8. IF the Embedding_Service is unavailable during a search request, THEN THE Backend SHALL return HTTP 503 with a descriptive error message.
9. IF the LLM is unavailable during a search request, THEN THE Backend SHALL return search results with AI_Reasoning set to a fallback message (e.g., `"Reasoning unavailable"`) rather than failing the entire request.

---

### Requirement 3: PII Unmasking

**User Story:** As an authenticated recruiter, I want to reveal a candidate's contact details, so that I can reach out to them directly.

#### Acceptance Criteria

1. WHEN an authenticated recruiter calls `POST /api/v1/recruiter/unmask` with a valid candidate identifier, THE Backend SHALL return the candidate's unmasked PII (name, phone, email).
2. WHEN a recruiter successfully unmasks a candidate's PII, THE Backend SHALL deduct 1 event from the recruiter's Event_Counter.
3. IF a recruiter's Event_Counter has reached the Free_Tier limit, THEN THE Backend SHALL return HTTP 402 and not reveal PII.
4. IF an unauthenticated user calls `POST /api/v1/recruiter/unmask`, THEN THE Backend SHALL return HTTP 401.
5. WHEN the same recruiter unmasks the same candidate more than once, THE Backend SHALL return the PII without deducting an additional event.

---

### Requirement 4: Event Tracking and Free Tier

**User Story:** As a recruiter, I want to know how many free actions I have remaining, so that I can manage my usage before a paywall is introduced.

#### Acceptance Criteria

1. THE Backend SHALL initialize each new authenticated recruiter's Event_Counter to 0 upon first sign-in.
2. WHEN an authenticated recruiter calls `GET /api/v1/recruiter/events`, THE Backend SHALL return the number of events consumed and the remaining Free_Tier allowance.
3. THE Backend SHALL read the Free_Tier limit from the `MAX_FREE_EVENTS` environment variable, defaulting to 10 if the variable is not set.
4. WHEN a resume download is initiated by an authenticated recruiter, THE Backend SHALL deduct 1 event from the recruiter's Event_Counter.
5. IF a recruiter's Event_Counter has reached the Free_Tier limit, THEN THE Backend SHALL block resume download requests and return HTTP 402.

---

### Requirement 5: Search Interface

**User Story:** As a recruiter, I want a simple, clean search page, so that I can quickly paste a job description and find candidates without friction.

#### Acceptance Criteria

1. THE Frontend SHALL display a single search input field with the placeholder text "Paste your JD here and we will take care of the rest."
2. WHEN a recruiter presses Enter or submits the search form, THE Frontend SHALL send the job description to the search endpoint and display results.
3. THE Frontend SHALL display a marketing banner with the text "Get the best candidates here — Free* (*till offer valid)" prominently on the search page.
4. WHILE a search is in progress, THE Frontend SHALL display a loading indicator to the recruiter.
5. IF the search returns no results, THEN THE Frontend SHALL display a message indicating no matching candidates were found.

---

### Requirement 6: Candidate Results Table

**User Story:** As a recruiter, I want to see a rich, ranked table of candidates, so that I can evaluate matches at a glance.

#### Acceptance Criteria

1. THE Frontend SHALL display search results in a table with the following columns: Rank, Role/Title, Skills, Location, Match Score, AI Reasoning.
2. WHEN results are displayed, THE Frontend SHALL show PII fields (name, phone, email) as masked placeholders by default.
3. WHEN a recruiter clicks to unmask a candidate's PII, THE Frontend SHALL prompt unauthenticated recruiters to sign in before proceeding.
4. WHEN an authenticated recruiter clicks to unmask a candidate's PII, THE Frontend SHALL call the unmask endpoint and display the revealed PII inline in the table.
5. THE Frontend SHALL display the recruiter's remaining event count when the recruiter is authenticated.
6. WHEN a recruiter's remaining events reach 0, THE Frontend SHALL disable the unmask and resume download actions and display a message indicating the free tier limit has been reached.

---

### Requirement 7: Authentication Flow

**User Story:** As a recruiter, I want to sign in with Google, so that I can access additional searches and PII unmasking without creating a new account.

#### Acceptance Criteria

1. THE Frontend SHALL provide a "Sign in with Google" button using Supabase_Auth.
2. WHEN a recruiter completes Google OAuth sign-in, THE Frontend SHALL store the Supabase session token and include it in all subsequent authenticated API requests.
3. WHEN an unauthenticated recruiter attempts a second search, THE Frontend SHALL display a sign-in prompt without losing the current search results.
4. WHEN an unauthenticated recruiter attempts to unmask PII, THE Frontend SHALL display a sign-in prompt.
5. IF a Supabase_Auth session expires, THEN THE Frontend SHALL prompt the recruiter to re-authenticate before allowing further authenticated actions.

---

### Requirement 8: Extension Profile Save Button

**User Story:** As a candidate, I want a "Save Profile" button in the extension's Profile tab, so that I can explicitly share my profile with recruiters when I choose to.

#### Acceptance Criteria

1. WHERE `shareAnonymized=true` is set in the candidate's extension settings, THE Extension SHALL display a "Save Profile" button in the Profile tab.
2. WHEN a candidate clicks "Save Profile", THE Extension SHALL POST the candidate's profile to `POST /api/v1/profile/sync`.
3. WHEN the sync request succeeds, THE Extension SHALL display a confirmation message to the candidate.
4. IF the sync request fails, THEN THE Extension SHALL display an error message with the failure reason to the candidate.
5. WHEN `shareAnonymized=false`, THE Extension SHALL hide the "Save Profile" button.
